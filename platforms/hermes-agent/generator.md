# Hermes-agent Platform -- Generation Reference

## Generator Source

The core generation logic lives at `generators/claude-md.md` and `generators/features/*.md`.
For Hermes-agent, those same feature blocks are composed into a **`SYSTEM_PROMPT.md`** instead
of a `CLAUDE.md`. The difference is delivery mechanism, not methodology.

---

## What Hermes-agent Platform Generates

A Hermes-agent deployment produces:

### 1. SYSTEM_PROMPT.md (Context File)

The primary context file, loaded as the system prompt at the start of every conversation.
Composed from the same feature blocks as the Claude Code platform, with these adaptations:

- **No hook references** — hook automation sections are replaced with equivalent manual
  instructions ("at session start, read self/identity.md and self/goals.md")
- **No `/skill-name` commands** — skill invocations reference `prompts/` files instead
  ("paste the contents of `prompts/reduce.md` to extract insights from a source")
- **Tool-use instructions** — if the model is configured with file-access tools (e.g.,
  Open WebUI with workspace file access), tool names are referenced explicitly; otherwise
  the agent is instructed to write file contents directly in the conversation

Contains the same methodology as the Claude Code context file:
- Philosophy and operating principles
- Discovery-first design
- Note design patterns (titles as claims, composability test, link semantics)
- Processing pipeline (described as manual invocations)
- Maintenance and self-evolution guidance
- Session rhythm (orient, work, persist)
- Ethical guardrails

Generated by: `generators/claude-md.md` with hermes-agent adaptations (see below).

### 2. prompts/ (Skill Prompt Files)

Plain markdown files, one per processing skill, that the user pastes into the chat to
invoke that workflow. These are the Hermes-agent equivalent of Claude Code's `.claude/skills/`.

Structure:
```
prompts/
├── reduce.md        -- Extract insights from a source file
├── reflect.md       -- Find connections between notes
├── reweave.md       -- Update older notes with new context
├── verify.md        -- Schema + description + link quality check
├── validate.md      -- Schema compliance check
├── next.md          -- Next-action recommendation
├── stats.md         -- Vault metrics summary
└── graph.md         -- Graph analysis and navigation
```

Each file contains the full skill instructions, vocabulary-transformed for the user's domain.
The user invokes a skill by pasting the file's content into the chat, optionally followed by
the target file or task.

Source: `skill-sources/*/SKILL.md` (vocabulary-transformed, stripped of Claude Code frontmatter).

### 3. ops/config.yaml

Same human-editable configuration as the Claude Code platform. Controls processing depth,
pipeline chaining, and feature toggles.

### 4. .hermes-agent/ (Platform Marker)

An empty directory (or minimal config file) that signals to the setup wizard that this vault
targets the Hermes-agent platform. Analogous to `.claude/` for Claude Code.

```
.hermes-agent/
└── config.yaml      # optional; platform-specific hints
```

Minimal `.hermes-agent/config.yaml`:
```yaml
# Ars Contexta -- Hermes-agent platform config
platform: hermes-agent
model: hermes3:8b-llama3.1-q5_K_M   # or your local model name
context_window: 131072               # tokens, from `ollama show <model>`
tool_use: false                      # set true if Open WebUI tools are configured
```

## Hermes-agent Context File Composition

When the setup wizard detects platform = hermes-agent, it composes `SYSTEM_PROMPT.md` using
the standard feature block composition algorithm from `skills/setup/SKILL.md` (Step 3), with
the following platform-specific substitutions:

### Hook → Manual Instruction Substitution

| Claude Code artifact | Hermes-agent equivalent |
|---------------------|------------------------|
| `session-orient.sh` hook on SessionStart | Context file instruction: "At the start of every conversation, read `self/identity.md`, `self/goals.md`, and `ops/reminders.md`." |
| `validate-note.sh` hook on PostToolUse | Context file instruction: "After writing any note, verify it has `description:` and `topics:` frontmatter before the session ends." |
| `session-capture.sh` hook on Stop | Context file instruction: "Before closing the conversation, write a session summary to `ops/sessions/YYYYMMDD-HHMMSS.md` with what was done, what changed, and what comes next." |
| `auto-commit.sh` hook on PostToolUse | Context file instruction: "After a productive session, run `git add -A && git commit -m 'Session: [brief description]'`." |

### Skill Reference Substitution

Replace all `/command-name` skill invocations with prompt-file references:

| Claude Code skill reference | Hermes-agent equivalent |
|----------------------------|------------------------|
| "Run `/reduce` on this source" | "Paste `prompts/reduce.md` into chat to extract insights" |
| "Chain to `/reflect` automatically" | "After reduce, paste `prompts/reflect.md` to find connections" |
| "`/next` for recommendations" | "Paste `prompts/next.md` to get next-action recommendations" |

### Context Window Adaptation

Check the actual context window with `ollama show <model>` (look for `context length`). The
generated context file includes explicit budget guidance regardless of window size, because
even large context windows benefit from disciplined budget management — attention quality
degrades as context fills, particularly for smaller models:

```markdown
## Context Budget

Manage context deliberately each session:

- **Orient phase**: Read self/ (3 files, ~1–2K tokens). Stay under 10% of context.
- **Work phase**: Process one source or one set of notes per session. Stay under 70%.
- **Persist phase**: Write session summary and update goals. Reserve 20%.

If a task needs more context than fits, split it. Use ops/queue/ to hand off between sessions.
One task per session is not a limitation — it is the design.
```

### Vocabulary transformation

Apply the same vocabulary transformation algorithm as the Claude Code platform. Domain-native
terms replace universal terms throughout `SYSTEM_PROMPT.md` and all `prompts/` files.

## Platform Characteristics

| Aspect | Detail |
|--------|--------|
| Context loading | `SYSTEM_PROMPT.md` loaded manually as system prompt |
| Hook language | None — replaced by context file instructions |
| Hook events | None — orient/validate/persist are manual |
| Skill format | Plain markdown in `prompts/` (no frontmatter required) |
| Agent format | `self/` directory (identity, methodology, goals) |
| File access | Via Ollama tool-use or model's direct file write capability |
| MCP support | Optional — via Open WebUI MCP configuration |
